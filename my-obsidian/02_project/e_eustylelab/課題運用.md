## 基本的なルール

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E5%9F%BA%E6%9C%AC%E7%9A%84%E3%81%AA%E3%83%AB%E3%83%BC%E3%83%AB)

- 《必須》課題はすべて [Backlog](https://eustylelab.backlog.com/) で管理してください。
- 《必須》GitHub の Issue は使用しないでください。
- 《必須》あらゆるタスクについて、可能な限り Backlog に課題を作成するようにしてください。
    - 外部から依頼や問い合わせがあった時。
    - 開発・調査・運用などのタスクが発生した時。
    - 将来対応すべき（かもしれない）課題に気付いた時。
- 《必須》緊急で作業を依頼された場合なども、落ち着いてまずは課題を登録してください。

「対応するかもしれない」レベルで、どんどん課題を登録して構いません。

## 各項目の入力・変更基準

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E5%90%84%E9%A0%85%E7%9B%AE%E3%81%AE%E5%85%A5%E5%8A%9B%E5%A4%89%E6%9B%B4%E5%9F%BA%E6%BA%96)

### 種別

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E7%A8%AE%E5%88%A5)

- 《必須》以下より適切なものを選ぶようにしてください。

|種別|内容|
|:--|:--|
|依頼・問い合わせ|外部から（または外部へ）の依頼や問い合わせについての管理。|
|開発：機能追加・改善|機能追加・改善・改修などの開発タスク。|
|開発：不具合修正|不具合対応で、リリースが必要となる開発タスク。|
|運用|チームやプロジェクト、インフラなどに関する運用系タスク。|
|調査|開発が発生するかどうかわからない調査系タスク。|

### 件名

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E4%BB%B6%E5%90%8D)

- 《必須》具体的な課題（＝タスク）の内容や、目的（＝ゴール）がわかるように心掛けてください。
- 《必須》タスクの種類やプロジェクト等を示すプレフィクスやサフィックスはなるべく付けないようにしてください。

```
よい例）予定一覧機能を追加する
悪い例）予定一覧機能
悪い例）【新システム】予定一覧機能を追加する
```

## 本文

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E6%9C%AC%E6%96%87)

- 《推奨》課題の「種別」ごとにテンプレートを設定していますので、その内容に従って入力してください。
- 《推奨》できる限りテンプレートの全項目に入力してください。
- 《任意》記入しない項目のタイトル等は削除して構いません。
- 《任意》テンプレートにない項目で、特に必要と思われる項目は追加して構いません。

## 優先度

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E5%84%AA%E5%85%88%E5%BA%A6)

- 《必須》下記の通り設定してください。
    - 対応が必要であるとわかっている場合 → 優先度を **高** に設定。
    - 対応が必要になるかもしれない場合 → 優先度を **低** に設定。
    - 優先度を「高」にすべきか「低」にすべきか判断できない場合 → 優先度を **中** に設定。

## 開始日

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E9%96%8B%E5%A7%8B%E6%97%A5)

- 《必須》着手する日が決まっている・わかっている場合はその日付を設定してください。 着手する日がわからない場合は空欄で構いません。
    - 課題を作成する日＝着手日の場合はその当日の日付を設定。
    - 着手する日が決まっている場合は《優先度》が「高」になるはずです。担当者の設定も忘れずに。
- 《推奨》着手する時にはなるべく設定するようにしてください。

## 期限日

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E6%9C%9F%E9%99%90%E6%97%A5)

### 「状態」が「未対応」の場合

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E7%8A%B6%E6%85%8B%E3%81%8C%E6%9C%AA%E5%AF%BE%E5%BF%9C%E3%81%AE%E5%A0%B4%E5%90%88)

- 《必須》**担当者が決まっている** 場合は **課題を完了すべき期限** を設定してください。
- 《推奨》**担当者が未定・決まっていない** 場合は **担当者を決めるべき期限** を設定してください。

### 「状態」が「処理中」以降の場合

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E7%8A%B6%E6%85%8B%E3%81%8C%E5%87%A6%E7%90%86%E4%B8%AD%E4%BB%A5%E9%99%8D%E3%81%AE%E5%A0%B4%E5%90%88)

- 《必須》**期限日が決まっている** 場合はその日付を設定してください。
- 《推奨》**期限日が特に決まっていない** 場合はおおよその作業完了日を見積もって設定してください。

## ○○をお知らせしたいユーザー

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E3%82%92%E3%81%8A%E7%9F%A5%E3%82%89%E3%81%9B%E3%81%97%E3%81%9F%E3%81%84%E3%83%A6%E3%83%BC%E3%82%B6%E3%83%BC)

- 《推奨》課題の追加や更新、コメントをお知らせしたいユーザーを選択してください。
    - 例えばこんな人を選んでください。
        - 開発課題で、マージやレビューを依頼する予定の人
        - 依頼・問い合わせから派生した開発課題で、依頼・問い合わせの対応を担当している人
        - 課題にコメントをくれた人
    - お知らせが多すぎると対応が大変になります。あまり多くの人に送りすぎないよう、気をつけてください。
    - かといってお知らせを送らな過ぎても、情報の共有が機能しなくなります。加減が大事です。

### 外部から依頼や問い合わせがあった時

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E5%A4%96%E9%83%A8%E3%81%8B%E3%82%89%E4%BE%9D%E9%A0%BC%E3%82%84%E5%95%8F%E3%81%84%E5%90%88%E3%82%8F%E3%81%9B%E3%81%8C%E3%81%82%E3%81%A3%E3%81%9F%E6%99%82)

#### 課題の作成・更新

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E8%AA%B2%E9%A1%8C%E3%81%AE%E4%BD%9C%E6%88%90%E6%9B%B4%E6%96%B0)

- 《必須》新しい依頼や問い合わせの場合は「種別」が「依頼・問い合わせ」の課題を新しく作成してください。
- 《必須》既存の依頼や問い合わせに関するものの場合は元の課題にコメントしてください。
- 《必須》担当者は常に「その依頼や問い合わせについて外部と遣り取りをする人」としてください。

#### 着手したら

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E7%9D%80%E6%89%8B%E3%81%97%E3%81%9F%E3%82%89)

- 《必須》依頼・問い合わせへの対応を開始する際に「状態」を **「処理中」** としてください。
    - 《推奨》依頼や問い合わせに応じて、開発や調査等の作業が発生する場合は、子課題を作成して対応してください。
- 《必須》開発や調査が発生する場合は、別途課題を作成してください。子課題にしても構いません。
    - 開発や調査を別の人に依頼する場合は、作成した別の課題の方の担当者をその人にしてください。

#### 返答したら

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E8%BF%94%E7%AD%94%E3%81%97%E3%81%9F%E3%82%89)

- 《必須》依頼・問い合わせへの回答、報告等を行ったら「状態」を **「返答待ち」** としてください。
- 《推奨》その際、返答内容を簡単にコメントするようにしてください。

#### 依頼元・問い合わせ元からの返答があったら

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E4%BE%9D%E9%A0%BC%E5%85%83%E5%95%8F%E3%81%84%E5%90%88%E3%82%8F%E3%81%9B%E5%85%83%E3%81%8B%E3%82%89%E3%81%AE%E8%BF%94%E7%AD%94%E3%81%8C%E3%81%82%E3%81%A3%E3%81%9F%E3%82%89)

- 《必須》回答・報告に対して依頼元・問い合わせ元からの返答があった場合は次のようにしてください。
    - 返答内容をコメントしてください。
    - 返答により、追加の作業が発生する場合は「状態」を **「要回答」** に戻してください。
    - 返答により、作業が発生しない＝依頼・問い合わせが完了する場合は「状態」を **「完了」** としてください。

### 開発（機能追加・改善・不具合修正）を行う時

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E9%96%8B%E7%99%BA%E6%A9%9F%E8%83%BD%E8%BF%BD%E5%8A%A0%E6%94%B9%E5%96%84%E4%B8%8D%E5%85%B7%E5%90%88%E4%BF%AE%E6%AD%A3%E3%82%92%E8%A1%8C%E3%81%86%E6%99%82)

#### 課題の作成・更新

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E8%AA%B2%E9%A1%8C%E3%81%AE%E4%BD%9C%E6%88%90%E6%9B%B4%E6%96%B0-1)

- 《必須》開発の内容に応じて、次の通り種別を選択し、課題を作成してください。
    - 新しい機能を追加したり、既存機能の改善・改修を行う（または行うかもしれない）場合は「開発：機能追加・改善」。
    - 不具合の修正を行う（または行うかもしれない）場合は「開発：不具合修正」。
- 《必須》担当者は「実際に開発を行う人」としてください。未定の場合は未設定のままとしてください。
- 《推奨》開発の規模が大きい場合（何日もかかる場合）は作業を分割し、子課題を作成するように心掛けてください。
    - 子課題を作成する場合、その子課題の担当者は別の人でも構いません。
    - 子課題の作成は、随時行ってください。

#### 着手したら

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E7%9D%80%E6%89%8B%E3%81%97%E3%81%9F%E3%82%89-1)

- 《必須》開発に着手する際「状態」を **「処理中」** としてください。

#### プルリクエストを作成（または更新）したら

[](https://github.com/eustylelab/readme/blob/master/docs/guidelines/issue-guideline.md#%E3%83%97%E3%83%AB%E3%83%AA%E3%82%AF%E3%82%A8%E3%82%B9%E3%83%88%E3%82%92%E4%BD%9C%E6%88%90%E3%81%BE%E3%81%9F%E3%81%AF%E6%9B%B4%E6%96%B0%E3%81%97%E3%81%9F%E3%82%89)

- 《必須》プルリクエスト作成時に課題の番号を記入するようにしてください。
- 《必須》課題を次の通り【編集】してください。
    - 「状態」を **「処理済み」** としてください。
    - 《本文》の「プルリクエストURL」欄にプルリクエストの URL を記載してください。
    - 「担当者」をマージ担当者（またはレビュー担当者）に変更してください。
    - 「コメント」に「レビューよろしくお願いします」的なメッセージを入れてもいいかもしれません。

現在 GitHub と Backlog の連携により、プルリクエスト作成時に課題番号に言及することで 課題の編集は自動的に行われるようになっています。

#### プルリクエストに修正依頼（Request Changes）をしたら
